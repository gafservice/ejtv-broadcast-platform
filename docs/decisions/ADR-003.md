# ADR-003

> *"Toda la plataforma debe compartir una única referencia temporal para garantizar la coherencia de los eventos, los registros del sistema y los servicios multimedia."*

# 1. Contexto

La plataforma **EJTV Broadcast Platform** integra múltiples servicios que requieren una referencia temporal consistente para su correcta operación, incluyendo servicios multimedia, registros del sistema, herramientas de administración, scripts de mantenimiento y futuras aplicaciones.

Durante el diseño de la infraestructura surgió la necesidad de definir cuál sería la fuente oficial de tiempo utilizada por todos los componentes de la plataforma.

Las alternativas consideradas incluían implementar un mecanismo propio de sincronización, utilizar servicios NTP adicionales o adoptar la infraestructura temporal proporcionada por el propio sistema operativo.

---

# 2. Problema

La utilización de múltiples fuentes de tiempo o la implementación de mecanismos independientes de sincronización incrementaría la complejidad del sistema y podría generar inconsistencias entre registros, eventos y servicios.

Era necesario establecer una única referencia temporal para toda la plataforma que garantizara coherencia, simplicidad de mantenimiento y estabilidad operativa.

---

# 3. Alternativas consideradas

## Alternativa A

Implementar un mecanismo propio de sincronización horaria.

### Ventajas

* Control completo sobre el proceso de sincronización.
* Posibilidad de incorporar lógica específica de la aplicación.

### Desventajas

* Mayor complejidad de desarrollo.
* Incremento del mantenimiento.
* Duplicación de funcionalidades ya disponibles en el sistema operativo.
* Mayor probabilidad de inconsistencias.

---

## Alternativa B

Instalar un servicio NTP adicional, como Chrony o ntpd.

### Ventajas

* Amplias opciones de configuración.
* Funcionalidades avanzadas para infraestructuras complejas.

### Desventajas

* Mayor complejidad administrativa.
* Componentes adicionales para mantener.
* Beneficio limitado para los requerimientos actuales de la plataforma.

---

## Alternativa C

Utilizar el reloj del sistema operativo sincronizado mediante `systemd-timesyncd`.

### Ventajas

* Integración nativa con Ubuntu Server.
* Menor complejidad.
* Bajo mantenimiento.
* Una única referencia temporal para todos los procesos.
* Compatibilidad con todos los servicios de la plataforma.

### Desventajas

* Menor flexibilidad respecto a soluciones especializadas.
* Dependencia de la infraestructura de sincronización del sistema operativo.

---

# 4. Decisión

Se establece que **EJTV Broadcast Platform** utilizará como referencia temporal oficial el **reloj del sistema operativo (System Clock)** mantenido por Linux y sincronizado mediante el servicio nativo **systemd-timesyncd**.

Todos los componentes de la plataforma deberán utilizar esta referencia temporal común, evitando la implementación de mecanismos independientes para la administración del tiempo.

---

# 5. Justificación

Ubuntu Server proporciona una infraestructura de sincronización estable, ampliamente probada e integrada con `systemd`.

Adoptar esta infraestructura permite:

* reducir la complejidad del sistema;
* evitar la duplicación de funcionalidades;
* garantizar una referencia temporal única;
* simplificar el mantenimiento;
* asegurar la coherencia entre todos los servicios.

Esta decisión sigue el principio de reutilizar servicios nativos del sistema operativo cuando satisfacen plenamente los requisitos del proyecto.

---

# 6. Consecuencias

La adopción de esta decisión implica que:

* MediaMTX utilizará el reloj del sistema.
* FFmpeg utilizará el reloj del sistema cuando corresponda.
* Los registros del sistema compartirán la misma referencia temporal.
* Los scripts de mantenimiento utilizarán la misma referencia temporal.
* Las futuras aplicaciones deberán utilizar el reloj del sistema operativo como única referencia temporal.

No se desarrollarán mecanismos propios de sincronización mientras los requerimientos de la plataforma permanezcan dentro del alcance actual.

---

---

# 8. Riesgos

La principal dependencia corresponde al correcto funcionamiento de `systemd-timesyncd` y a la disponibilidad de servidores NTP confiables.

Este riesgo se considera bajo debido a la estabilidad de la implementación nativa de Ubuntu Server y a que el servicio forma parte del sistema operativo.

---

# 9. Seguimiento

Esta decisión permanecerá vigente mientras la plataforma no requiera una infraestructura de sincronización de alta precisión o un servidor NTP propio.

Cualquier cambio futuro deberá documentarse mediante un nuevo **Architecture Decision Record**, preservando la trazabilidad de las decisiones arquitectónicas.

---

# Conclusión

La plataforma **EJTV Broadcast Platform** adopta el reloj del sistema operativo sincronizado mediante **systemd-timesyncd** como única referencia temporal para todos sus componentes.

Esta decisión simplifica la arquitectura, reduce el mantenimiento y garantiza que todos los servicios compartan una referencia temporal consistente, favoreciendo la estabilidad y la trazabilidad de la plataforma a largo plazo.
